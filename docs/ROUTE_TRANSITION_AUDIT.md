# Аудит переходов между маршрутами

**Дата:** 2026-04-01  
**Цель:** понять, почему навигация ощущается тяжёлой, и что оптимизировать без смены дизайна.

## Почему переходы сейчас ощущаются медленными

1. **Крупные клиентские границы на общих компонентах** — `Navbar` и `NonHomeMobileHeader` целиком помечены `"use client"`, хотя интерактивны только кнопка меню, оверлей и блокировка скролла. При каждом переходе на страницу с этим кодом гидратируется весь компонент вместе с разметкой логотипа и (для главной) десктопной навигации, которая могла бы остаться серверной.

2. **Тяжёлые чанки при входе на маршрут** — `/portfolio` и `/services` синхронно подтягивают большие desktop-сцены (`PortfolioDesktopScene`, `ServicesDesktopScene` + деревья Figma-слоёв). Пока парсится и выполняется весь модуль, первый кадр после клика по ссылке может запаздывать.

3. **Нет лёгкого UI при переходе** — в сегментах нет `loading.tsx`: при медленной сети пользователь видит задержку без обратной связи.

4. **Общий layout лёгкий** — `app/layout.tsx` не оборачивает приложение в лишние client-провайдеры; узкое место в основном в страницах и shared nav.

5. **Prefetch** — `Link` из Next.js по умолчанию префетчит видимые ссылки в продакшене; явная настройка не критична, но стоит не отключать prefetch на основной навигации.

## Какие маршруты тяжелее всего

| Маршрут                 | Наблюдение                                                                                                    |
| ----------------------- | ------------------------------------------------------------------------------------------------------------- |
| `/portfolio`            | Большая desktop-сцена в `CanvasScaler`, много слоёв/картинок; мобильная часть легче.                          |
| `/services`             | Аналогично: desktop canvas + декоративные блоки.                                                              |
| `/`                     | Динамический `NeetrinoHome` уже отделён; мобильная ветка тянет `Navbar` (client целиком — кандидат на split). |
| `/about-us`, `/contact` | Относительно проще: хедер + контент; форма контактов остаётся client по необходимости.                        |

## Какие shared-компоненты удорожают навигацию

- **`Navbar`** — весь файл client; десктопные ссылки и логотип не требуют клиента.
- **`NonHomeMobileHeader`** — весь client; логотип и оболочка хедера могут быть server, интерактив — маленький shell.
- **`CanvasScaler`** — должен оставаться client (resize); оборачивается только desktop-веткой.
- **`Footer` / `FooterMessageForm`** — уже разделены; ок.

## Какие файлы планируется оптимизировать

- `components/sections/Navbar.tsx` — server + выделенный `NavbarMobileShell` (client).
- `components/sections/NonHomeMobileHeader.tsx` — server + `NonHomeMobileShell` (client) с `children` для логотипа.
- `lib/nav-links.ts` — общие ссылки навигации.
- `lib/hooks/useBodyScrollLock.ts` — общая блокировка скролла.
- `app/portfolio/page.tsx` — `dynamic()` для `PortfolioDesktop` с лёгким placeholder.
- `app/services/page.tsx` — `dynamic()` для desktop-блока (обёртка `ServicesDesktopCanvas`).
- `app/portfolio/loading.tsx`, `app/services/loading.tsx`, при необходимости `about-us` / `contact` — минимальная полоска/фон без тяжёлых скелетонов.

## Что визуально не меняем

- Внешний вид хедеров, оверлеев, ссылок и порядок блоков.
- Тексты, маршруты, Figma-координаты и декоративные слои (кроме отложенной загрузки JS desktop-сцен).
- Поведение мобильного меню (открытие/закрытие, блокировка скролла).
